# Sector Research Workflow 重构分模块验证计划

## Summary
- 当前已完成：`run_sector_stage` 统一入口、`workflow_stages.yaml` 阶段策略、`evidence_collect` source manifest 阶段、`evidence_draft` evidence skeleton 阶段、相关 skill 文档同步。
- 当前进入：模块级验证与调试，不继续扩展新功能。
- 调试节奏：每个模块失败后只在该模块范围内修复并复测，通过后再进入下一个。

## Module Validation Order
1. **Baseline / Safety Check**
   - 只读确认当前 git 状态、正式目录 card 数量、`00_总表/99_日志` 不存在、项目 loader 可运行。
   - 目标：确认后续验证不会误判已有遗留文件。

2. **Stage Runner Core**
   - 验证 `run_sector_stage` 的 stage 枚举、参数转发、policy 加载、warning-only 规则、`--write-summary`。
   - 通过标准：`scope_check`、非法 stage、缺必填参数、warning-only 处理都符合预期。

3. **Evidence Collect**
   - 验证 preview 默认不写文件；`--write-manifest` 只写 raw source manifest。
   - 验证本地 PDF、已有 `.txt`、Tushare JSON cache 的记录生成。
   - 通过标准：不写 active evidence、不碰正式目录、不改 manifest/sector binding。

4. **Evidence Draft**
   - 验证 `source_manifest -> draft YAML skeleton`。
   - 验证 preview 默认不写；`--write-draft` 只写 audits/evidence_drafts。
   - 通过标准：草稿含 `DRAFT_PLACEHOLDER/TODO_MANUAL_EXTRACTION`，且不能被误注册为 active evidence。

5. **Evidence Register**
   - 只用已 curated 的 active evidence 文件做 dry-run 和 apply 测试。
   - 验证不会手工改 `sector_universe.yaml`，不会重复绑定，不会跨 sector 绑定。
   - 通过标准：dry-run 可预览；apply 后 binding/schema audit 通过。

6. **Evidence Gate**
   - 验证 binding/schema/coverage 三联 gate。
   - 验证 coverage 对目标 sector OK 时，其他 P0/P1 MISSING 仍作为非阻塞基线。
   - 通过标准：ERROR=0；目标 sector coverage 解析准确。

7. **Candidate / Publish Gate Wrappers**
   - 验证 `generate_candidate`、`candidate_gate`、`publish_gate` dry-run。
   - 不执行正式发布，除非用户另行明确确认。
   - 通过标准：candidate 留在 audits；publish_gate 只生成 dry-run manifest/readiness，不写正式目录。

8. **Skills / Docs Consistency**
   - 验证 `sector-research-orchestrator`、`evidence-miner`、`quality-auditor` 的说明与脚本实际行为一致。
   - 通过标准：skill quick_validate 通过，文档不再暗示跳过 source manifest/draft/registration 分层。

## Remaining Work
- 增加 `source metadata sidecar` 标准：补齐 CNINFO URL、公告日期、公司名、标题映射。
- 增加 `draft placeholder guard`：禁止含 `DRAFT_PLACEHOLDER/TODO_MANUAL_EXTRACTION` 的文件注册为 active evidence。
- 增加 `source_manifest -> curated evidence` 校验器：检查 excerpt、claim、metrics、missing_fields 是否已人工整理。
- 增加 Tushare cache splitter：把一个 JSON cache 拆成 income / daily_basic / daily 等标准 source/evidence 草稿。
- 后续再考虑 MinerU API 阶段化：提交、轮询、结果登记、解析文本绑定，但不放在当前第一轮模块验证里。

## Test Plan
- 每个模块固定执行：脚本编译、runner preview、必要的写入模式、正式目录复核。
- 每个模块完成后记录：命令、exit code、写入文件、未写入范围、ERROR/WARNING/INFO。
- 任一模块失败：停止后续模块，只在该模块内修复并复测。
- 每次涉及写入后复核：正式目录仍不新增总表、日志、source index、score table、投资建议。

## Assumptions
- 本轮只做重构验证，不发布新的 sector card。
- 使用项目 Conda Python：`C:\Projects\03_Investment\.conda\investment-system\python.exe`。
- 不批量删除任何文件或目录。
- 现有未跟踪文件视为前序遗留状态，不清理、不纳入当前模块验证的成功条件。

# 模块 7/8 详细重构与逐模块验证计划

## Summary
- 模块 7 聚焦：把 `Generate Candidate -> Candidate Gate -> Publish Gate -> sector-card-only publish -> Post-publish Check` 的脚本包装、边界检查、报告语义调稳。
- 模块 8 聚焦：把 skills/docs 与脚本行为对齐，让后续 Codex 不再靠上下文记忆判断流程。
- 执行节奏：模块 7 全部调试通过后，才进入模块 8；任一子项失败，只在当前模块内修复并复测。

## 模块 7：Candidate / Publish Gate Wrappers

### 目标
- 让候选生成、候选门禁、发布预演、正式发布、发布后检查都能通过 `run_sector_stage.py` 一致调用。
- 保持简化链路，不恢复 `formal candidate audit -> gated formal -> gated formal audit` 的旧链路。
- 确保正式发布仍只有 `sector_card_only`，且必须显式人工确认。

### 7.1 Candidate Wrapper 验证
- 验证 `run_sector_stage --stage generate_candidate` 只调用 `build_formal_candidate_outputs --candidate-only`。
- 统一检查 candidate 输出目录必须在 `audits/formal_candidate_outputs/`。
- 增加或确认 candidate metadata 记录：
  - `project_id`
  - `sector_id`
  - `candidate_only=True`
  - `source_ids`
  - `evidence_ids`
  - `NOT_RATED`
  - `NOT_INVESTMENT_ADVICE`
  - 输出文件路径
- 验收：
  - 不写 `科技主线调研输出/**`
  - 不生成 gated formal
  - 不生成 release manifest
  - 不生成正式评分或投资建议

### 7.2 Candidate Gate 硬化
- 在 `audit_formal_candidate_outputs` 中补强或确认以下硬检查：
  - source_id / evidence_id 必须能在 active evidence 中闭环
  - candidate 文件不得在正式输出目录
  - candidate 不得引用 draft evidence skeleton
  - 内容不得包含 `DRAFT_PLACEHOLDER`
  - 内容不得包含 `TODO_MANUAL_EXTRACTION`
  - 内容不得出现买入、卖出、建仓、加仓、减仓、清仓、目标价、仓位建议
  - 必须保留 missing / conflict / counter-evidence 段落
  - score 只能是 placeholder / not_applicable
- 验收：
  - 正常 candidate：ERROR=0
  - 含 draft placeholder 的 candidate：必须 ERROR
  - 引用不存在 source_id/evidence_id：必须 ERROR
  - 出现投资建议：必须 ERROR

### 7.3 Publish Gate Dry-run 硬化
- 验证 `run_sector_stage --stage publish_gate` 只做发布预演。
- `prepare_formal_publish --dry-run --publish-scope sector_card_only --no-overwrite` 必须生成 release manifest。
- `audit_formal_publish_readiness` 只检查发布动作，不重复做候选内容审计。
- release manifest 必须记录：
  - `publish_scope=sector_card_only`
  - source candidate path
  - source hash
  - target formal path
  - target exists / overwrite risk
  - excluded outputs no-action
  - no total table / no log / no score table / no source index
- 验收：
  - target 不存在时：Publish Gate 可 PASS
  - target 已存在时：Publish Gate 必须 ERROR 或明确阻断
  - file_map 只能包含 `sector_card`
  - excluded outputs 不得有 publish_action

### 7.4 sector-card-only Publish Wrapper
- 验证 `run_sector_stage --stage publish_sector_card_only` 必须带 `--confirm-publish`。
- 确认 publish 脚本拒绝：
  - 非 `sector_card_only`
  - 未带 `--no-overwrite`
  - target 已存在
  - source hash 变化
  - target 路径落入总表或日志目录
- 发布日志必须记录：
  - source candidate file
  - target formal file
  - source hash / target hash
  - publish time
  - overwrite=False
  - gates_passed=True
  - investment_advice=False
  - forbidden_artifacts_created=False
- 验收：
  - 无确认参数：必须失败
  - target 已存在：必须失败
  - 正常发布只写 1 张 sector card

### 7.5 Post-publish Check
- 验证 `run_sector_stage --stage post_publish_check` 串联：
  - `audit_formal_publish_result`
  - `validate_outputs`
  - `audit_pipeline_readiness`
- 检查发布后：
  - source hash == target hash
  - 正式目录 card 数量符合预期
  - `00_总表` 不存在或未被本阶段写入
  - `99_日志` 不存在或未被本阶段写入
  - 未生成 source index / missing conflict log / comparison table / score table
  - 未生成投资建议
- 验收：
  - publish log 存在且可追溯
  - readiness `BLOCKER=0,HIGH=0`
  - validate_outputs exit 0

### 7.6 模块 7 测试顺序
1. `py_compile` candidate/publish 相关脚本。
2. 用已存在 candidate 跑 `candidate_gate` 正常样例。
3. 构造或复用只读异常样例，验证 placeholder / 投资建议 / 缺 source 闭环会失败。
4. 跑 `publish_gate` dry-run。
5. 对已存在 target 验证 no-overwrite 阻断。
6. 不执行正式发布，除非用户单独确认。
7. 输出模块 7 验证报告，列出每条命令 exit code。

## 模块 8：Skills / Docs Consistency

### 目标
- 让 skill 文档成为可执行流程导航，而不是泛泛说明。
- 不新建主题型 skill，不创建 `cpo-skill`、`pcb-skill` 之类一次性 skill。
- 保持 workflow-stage skill 分工：orchestrator 协调，evidence-miner 管证据，quality-auditor 管 gate。

### 8.1 sector-research-orchestrator
- 对齐最终标准流程：
  - `scope_check`
  - `evidence_collect`
  - `evidence_draft`
  - 人工 curate
  - `evidence_register`
  - `evidence_gate`
  - `generate_candidate`
  - `candidate_gate`
  - `publish_gate`
  - 人工确认
  - `publish_sector_card_only`
  - `post_publish_check`
- 明确哪些阶段默认 preview / dry-run。
- 明确哪些阶段允许写入，哪些阶段禁止正式输出。
- 验收：
  - SKILL.md 不再暗示手工编辑 `sector_universe.yaml`
  - 不再暗示 candidate 可直接发布
  - 不再暗示 broad legacy runner 是默认路径

### 8.2 evidence-miner
- 明确三层证据流：
  - source manifest：原始材料索引
  - evidence draft：待人工摘录骨架
  - active evidence：已校验并注册的正式证据 YAML
- 明确 draft 不可用于 candidate。
- 明确 `DRAFT_PLACEHOLDER/TODO_MANUAL_EXTRACTION` 是阻断标记。
- 验收：
  - evidence-miner 不负责写正式 card
  - evidence-miner 不绕过 evidence_register
  - evidence-miner 不把 context-only 当 strong evidence

### 8.3 quality-auditor
- 明确 gate 顺序和职责：
  - Evidence Gate 查 binding/schema/coverage
  - Candidate Gate 查内容质量、source/evidence 闭环、missing evidence、无投资建议
  - Publish Gate 查发布路径、no-overwrite、写入范围
  - Post-publish Check 查 hash、正式目录、禁生成物
- 增加或确认 placeholder leakage 规则：
  - draft placeholder 不得进入 candidate
  - debug placeholder 不得进入 research-grade 输出
- 验收：
  - quality-auditor 不重复承接 evidence mining
  - 不建议直接运行旧 broad validation 作为默认路径

### 8.4 references 文档
- `workflow.md` 保持流程说明，不塞过多脚本细节。
- `quality_gates.md` 保持 gate 验收条件。
- 必要时新增一个简短 reference，例如 `stage_runner.md`，只记录各 stage 的输入、输出、默认模式和允许写入范围。
- 验收：
  - SKILL.md 保持简洁
  - 详细流程放 references
  - 没有 README/CHANGELOG 等无关文档

### 8.5 模块 8 测试顺序
1. 运行 skill quick_validate：
   - `sector-research-orchestrator`
   - `evidence-miner`
   - `quality-auditor`
2. 抽查文档与 `workflow_stages.yaml` 是否一致。
3. 抽查文档中的命令是否存在于 `run_sector_stage`。
4. 用一条真实命令验证文档入口可运行，例如 `scope_check` 或 `evidence_gate`。
5. 输出模块 8 验证报告。

## Stop Conditions
- 任一模块出现脚本 ERROR 且不是预期失败样例，停止后续模块。
- 任一测试写入正式目录，立即停止。
- 任一脚本生成总表、日志、source index、score table，立即停止。
- 任一候选或发布流程出现投资建议、评级、目标价、仓位建议，立即停止。
- skill 文档与实际 stage 参数不一致，先修文档或脚本，不进入下一模块。

## Assumptions
- 模块 7 调试期间默认不正式发布。
- 模块 8 只修改已有 workflow-stage skills，不新增主题型 skill。
- 遇到失败采用“修到通过”节奏，但修复范围限定在当前模块。
- 前序遗留未跟踪文件不清理、不删除，只在报告中区分。
